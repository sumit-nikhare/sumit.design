Inventory

A facility inventory module built around operational truth: what’s in stock, what’s running out, what moved in/out, and who acknowledged it. The design supports day-to-day workflows like inward acceptance, outward issuing, and manual stock correction, while keeping oversight and accountability visible.

Module scope

What Inventory covers

  • Inventory summary: inward/outward counts, category-level stock status
  • Medicines catalog with tabs: In-stock, Out of stock, Few left, Inactive
  • Inward receipts: list → detail → view items → accept items / mark received
  • Outward issues: select institute → select items → issue qty → notify
  • Medicine details: available quantity + options (add quantity, request medicine, mark inactive)
  • Item ledger: Opening / Inward / Outward / Issued, with filters by date and type
  • Minimum indenting quantity to control “Few Left” threshold

What Inventory does not try to be

  • A procurement system (it supports receipt/issue and requests, not full tender cycles)
  • A purely administrative module (it’s integrated into daily facility work)
  • A single-role experience (actions differ by responsibility and scope)

Facility-scoped by default

Inventory surfaces the facility and geography context (example: Bangalore Urban 1, HWC(UPHC)), and inward/outward flows reference “issued from” and “issued to” institute types (PHC, sub-center). This makes scope explicit and reduces accidental cross-facility actions.

Problem framing

Inventory fails when it’s treated as paperwork. Facilities need a system that supports fast acknowledgement of receipts, controlled issuance, and believable stock levels. If inventory is wrong, OPD and pharmacy workflows become theatre.

Receipts must be acknowledged

Inward isn’t “received” until items are accepted. The UI needs explicit states like “Not received” vs “Accepted”.

Issuing needs a trace

Outward needs “issued to” institute selection and clear quantities per item, plus notification.

Stock status must be actionable

“Few left” is only useful if it’s system-driven (min indent threshold) and tied to requests/indent flows.

Connectivity is a real constraint

The screens explicitly show “No Internet Found”. The system needs graceful failure states and recovery paths.

Users and RBAC

Inventory actions carry governance risk. The UI separates “view stock” from “change stock,” and treats receipts and issues as accountable actions tied to institutes and reference numbers.

Primary workflows

1) Inward receipt and acceptance

Inward is treated as a traceable record (inward number, sender warehouse/institute, receipt date), and items are accepted before the inward is considered complete (e.g., 174/174 accepted). “Mark all items as received” supports bulk acknowledgement.

  • Inward list grouped by date + received vs not received
  • Inward details: institute info + sender + view items count
  • Item list supports search (name/medicine code) + batch details
  • Bulk receive for high-volume indents

2) Outward issue to another institute

Outward is a controlled issuance flow: select destination institute, select items, set issue quantities, and notify the receiver (example: “Notify to HWC”, “Notified”). Pending receipt status is visible.

  • Outward list shows “received on” vs “not received”
  • Outward details include issued-from/issued-to institute types
  • Item selection supports partial issuing + save
  • Receiver acknowledgement can remain pending

3) Stock browsing and status triage

Medicines are browseable with search, and stock status is surfaced as tabs: In-stock, Out-of-stock, Few Left, Inactive. This supports operational scanning without forcing users into item details every time.

  • Search by name or medicine code
  • Clear available quantity on list rows
  • Optional filters (including medicine type)

4) Medicine details + item ledger

Medicine details support “options” actions (add quantity, request medicine, mark inactive) and provide an auditable transaction log: Opening, Inward, Outward, Issued, with filters by date and type.

  • Ledger is filterable: date range + type (Inward/Outward/Issued)
  • Displays closing balance per transaction
  • Supports entries like “Manually Added”, “Purchased”, “Received from PHC”, “Issued to SC”, “Issued to patients”

5) Manual quantity updates with safeguards

Manual add/edit quantity exists for real-world corrections (private purchase, purchasing, reconciliation). The UI asks for minimum indenting quantity, which drives the system moving items into “Few Left” automatically.

  • “Add/Edit current quantity” supports corrections
  • Minimum indenting quantity defines when it becomes “Few Left”
  • Options include “Request medicine” for resupply workflows

Key design decisions

Treat stock movement as the system, not a spreadsheet

The ledger view (Opening/Inward/Outward/Issued + closing balance) makes the question “why is stock like this” answerable without exports.

Bulk receive for high-volume indents

“Mark all items as received” reduces acknowledgement friction for large indents (hundreds of items), while still keeping an acceptance boundary.

Separate “Outward” from “Issued”

Outward to another institute and issuing to patients are not the same thing. Treating them as distinct transaction types keeps reporting and audits credible.

Min indent quantity as a product rule

“Few Left” should not be manually curated. The min indent threshold makes it a predictable system behavior and supports reorder/request flows.

Connectivity failure state is explicit

The “No Internet Found” screen makes the constraint visible. Operational modules need honest states so staff don’t do repeated work blindly.

Make “scope” visible everywhere

Showing geography + facility context reduces accidental actions across sites and supports governance expectations in multi-institute hierarchies.

Reusable UI patterns

Status-as-navigation

In-stock / out-of-stock / few left / inactive tabs reduce cognitive load and support quick triage.

Detail page with “Options” sheet

Actions (add qty, request, mark inactive) are grouped, reducing accidental edits and keeping browsing fast.

Ledger filter pattern

Filter by date range and transaction type. Same pattern can be reused in OPD timelines or referrals.

“Acknowledge” as a first-class action

Inward and outward both contain a responsibility boundary: accepted vs pending. The UI makes this boundary explicit so responsibility doesn’t dissolve into “someone did it.”

Edge cases and controls

Inward date mismatch

The flow includes “No inward found” for a selected date. This prevents staff from force-fitting receipts into the wrong record.

Over-received vs indent count

Screens show cases where received items exceed indent items. The UI must handle exceptions without corrupting stock math.

Manual additions need provenance

Manual stock updates require a “mode” (purchased/private purchase/manually added). Without that, audits become fiction.

Outward acknowledgement pending

Outward records show “received date: pending”. The system needs to make pending visible and searchable to avoid silent loss.

Connectivity loss

“No Internet Found” must map to safe behavior: prevent partial inward acceptance or quantity edits from appearing successful when they aren’t.

How Inventory connects to OPD

Inventory is not a “separate module.” It directly determines whether prescribing and dispensing succeed. In OPD, medicines show stock availability. In pharmacy flows, dispensing becomes an accountable issuance.

At prescription time

Stock visibility reduces downstream negotiation and avoids writing prescriptions the facility cannot fulfill.

At dispensing time

Dispensing should create “Issued to patients” transactions, keeping stock and care records aligned.

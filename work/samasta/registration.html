Beneficiary registration with ABHA inclusion

A role-aware identity entry flow designed for PHC + field operations. The module prioritizes fast lookup, controlled creation, and optional ABHA linking with explicit consent and resilient fallbacks.

Problem

Identity entry must be fast, controlled, and trustable

Beneficiary registration is the entry point to every downstream workflow (OPD, outreach, inventory-linked dispensing). The interface must prevent duplicates, support role-based ownership, and still feel fast under field conditions.

What makes it hard

  • High duplicate risk: names and demographics repeat across villages; search must be encouraged before create.
  • Optional ABHA path: linking should be available without blocking service delivery when it can’t complete.
  • Consent + sensitivity: identity linking requires explicit consent and clear opt-out.
  • Connectivity constraints: online steps must have safe fallbacks to avoid data loss or repeated work.
  • RBAC reality: create/edit rights differ across ASHA/ANM/CHO/facility staff, and the UI must reflect it.

Design goal: keep the flow search-first, consent-aware, and recoverable under interruptions.

Users

Different roles create and reuse identity in different contexts

Registration is used by field workers creating beneficiary records during household visits and by facility teams who need quick identity retrieval for service delivery. The interface aligns the same flow to different responsibilities and permissions.

ASHA (Field)

Creates beneficiary records, captures core demographics, and initiates linking when feasible.

UX need: quick capture, resumable steps, low failure anxiety.

ANM / CHO / Facility staff

Reuses existing records for service delivery, verifies details, and handles edits that require more authority.

UX need: fast search, reliable match cues, clear edit boundaries.

Operational realities baked into the UI

  • Users switch contexts quickly (visit interruptions, shared devices).
  • Connectivity changes across locations, so online-only steps must be explicit.
  • Users need confidence that the right person was selected before creating new records.

Flow

Search → verify → create → optionally link ABHA

The flow starts with retrieval (search) and only moves to creation when a confident match is not found. ABHA inclusion sits as an optional branch that should never block registration completion.

Branch behavior (as shown in the diagram)

  • If ABHA is available: show consent, proceed with online verification/linking, then return to the beneficiary profile.
  • If ABHA is not available / user declines: complete registration normally and keep a clear option to link later.
  • If offline or validation fails: preserve form state and continue registration without forcing retries.

Design Decisions

Designed for speed without sacrificing governance

1) Search-first entry

The default path pushes users to search before create to reduce duplicates and increase confidence in reuse.

2) Progressive disclosure for ABHA inclusion

ABHA linking is presented as an optional branch that can be completed when conditions allow, rather than a hard gate.

3) Consent is a UX step, not a checkbox

Consent screens are explicit and placed before any sensitive linking action, with a clear “skip” path that doesn’t punish the user.

4) Recoverable failures

When online steps fail, the UI explains what happened and lets users continue registration without losing progress.

5) Role-aware edit boundaries

Create and edit capabilities differ by role; the interface avoids “show everything, fail later” behavior by shaping the UI around permissions.

Outcome: the flow stays usable in field conditions while protecting identity integrity and consent.

RBAC as UX

Permissions define what users can confidently complete

Beneficiary registration touches sensitive data and sets downstream eligibility. The UI treats RBAC as a first-class constraint by controlling visibility, editability, and action affordances.

Boundary 01

Create vs edit

Some roles can create records; edits to core identity fields can be restricted or audited.

Boundary 02

ABHA actions

Linking steps can be limited to authorized roles and require explicit user consent within the flow.

Boundary 03

Auditability

Where identity changes matter, actions are traceable so accountability survives handoffs.

RBAC reduces clutter for frontline roles and makes governance explicit where it matters.

Edge cases

Designed for failure modes that happen daily

Weak network during ABHA linking

The user can continue registration without losing progress. The UI makes “online required” steps explicit.

Possible match found

Match cues support verification so users reuse the correct record instead of creating a duplicate.

Consent declined

The system respects opt-out and completes registration normally, keeping “link later” available.

System fit

Where this module connects across Samasta

  • OPD: registration creates the identity anchor for encounters and prescriptions.
  • Outreach: field follow-ups depend on consistent beneficiary identity and contact data.
  • Dashboard: oversight and reporting reliability improves when duplicates are controlled at entry.

Registration is the platform’s front door. If it’s messy, everything downstream inherits the mess.

Impact

Cleaner identity records and less friction for daily work

  • Encourages record reuse through search-first entry and verification cues.
  • Supports ABHA inclusion without blocking registration completion.
  • Makes consent explicit and safe, improving user trust in identity linking.
  • Reduces downstream confusion in OPD and outreach by stabilizing beneficiary identity at the source.

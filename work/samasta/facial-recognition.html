Facial Recognition

A front-door identity flow that reduces friction at the facility entry point while keeping consent, fallbacks, and handoffs explicit. The module supports three paths: search by identifiers, face scan + online search, and new registration with optional biometric enrollment.

Module scope

What this module covers

  • Search people by Mobile Number, Name, or ABHA ID
  • Face scan workflow with guidance (lighting, background, distance)
  • Global/online search fallback when local search fails
  • Consent gating for face scan / biometric registration
  • Result selection → Purpose of Visit (OPD / New follow-up)
  • No-result → registration path with OCR-assisted form fill
  • Optional biometric enrollment (“Add facial biometric” vs “Don’t add”)
  • ABHA link/create prompts where relevant

What this module intentionally does not solve

  • Clinical workflow itself (that begins after Purpose of Visit)
  • Identity governance policy (consent language exists; policy sits above UX)
  • “Perfect match” guarantees (the UX supports verification and fallbacks)

Why this module is a platform concern

Identity sits upstream of OPD, follow-ups, outreach, and reporting. If identity is slow or inconsistent, every downstream module pays the cost through duplicates, rework, and mistrust in records. This flow was designed as a stable entry primitive: Find → Confirm → Proceed.

Problem framing

At facility entry points, identity is a throughput problem. Staff need to locate a beneficiary quickly, avoid duplicates, and transition into service flows without losing time. A biometric option helps, but only if consent is clear, failures are handled, and fallbacks remain available.

Searching needs multiple keys

Mobile, name, and ABHA ID reduce dependence on a single identifier and support partial recall in real-world interactions.

Biometrics require explicit consent

Consent is treated as a gating step (not a tooltip), with a “Don’t add” branch that still lets the user proceed.

Capture quality is a UX problem

Lighting, background, and distance guidance reduces “silent failure” and supports consistent capture across facilities.

No-result must still continue

When scan/search fails, the system offers online search and registration, avoiding dead ends that create duplicates later.

Users and RBAC

This flow sits on a responsibility boundary: searching and confirming identity is not the same as registering a new person or enrolling a biometric. The UI design keeps those actions separate so facilities can enforce governance without breaking throughput.

Primary workflows

1) Search by identifiers

Search supports Mobile Number, Name, and ABHA ID. Results show demographics and location details to help resolve ambiguity when multiple matches exist.

  • Search bar + results list with multiple candidates
  • Clear “No result found” state
  • Fallback to “Try online search” when local results fail

2) Face scan search (with online lookup)

Face scan includes capture guidance and online searching states. When successful, users can “View results” and proceed.

  • Capture checklist: no masks/glasses, single face, centered, clear background, adequate lighting
  • Real-time guidance: “Light good”, “Background not good”, “Move further”
  • System state: “Searching online…”
  • Outcomes: results list OR “No people found”

3) Consent gating

Consent is surfaced as a dedicated step with a clear acceptance action (“Give consent”) and an explicit opt-out path (“Don’t add”). This prevents biometric enrollment from becoming an implicit requirement to receive care.

  • Consent form appears before biometric actions
  • Users can proceed without enrolling biometrics
  • Consent supports ABHA-related context (link/create prompts)

4) No-result → registration (OCR-assisted)

When scan/search fails, the flow supports registering a new person. OCR scan helps prefill structured fields to reduce typing burden.

  • OCR scan → registration form fields (name, DOB/age, gender, mobile, address, geography)
  • Save and Next continues into downstream flows
  • Optional “Add facial biometric” after registration

5) Identity confirmed → Purpose of Visit

Once a beneficiary is selected, the next decision is operational: why are they here. The module hands off into visit flows via “Purpose of Visit” (OPD / New Follow-up), and surfaces ABHA status (Not created / Not linked) with link/create actions.

Key design decisions

Design for ambiguity (multiple matches)

Results lists include demographic and household context so staff can disambiguate candidates without opening multiple profiles.

Treat consent as a gate, not decoration

The module explicitly supports opting out (“Don’t add”) while still allowing care workflows to proceed.

Make capture quality visible in the UI

Real-time hints (lighting/background/distance) reduce repeated failures and normalize capture conditions across sites.

Always provide a “continue” path

No-result states route users to online search or registration rather than forcing them to restart or abandon the process.

Handoff into work, not into more identity

After identity confirmation, the next screen is operational (“Purpose of Visit”). This prevents identity steps from expanding into a maze.

Keep ABHA status visible at the decision point

ABHA not linked/created is surfaced alongside the beneficiary card so linking can happen intentionally, not buried later.

Edge cases and controls

Scan success, search fail

The flow distinguishes capture from match (“scan/search fail”), prompting online search or registration rather than repeating capture endlessly.

No people found (online)

“No people found” is a valid outcome with a clear next step. This prevents staff from creating duplicates through guesswork.

Multiple similar candidates

When multiple results share family/spouse fields and location, the UI relies on demographic context and structured identifiers to avoid wrong selection.

Consent refused

The flow supports continuing without biometric enrollment. This protects access and keeps the module compliant with real-world opt-out behavior.

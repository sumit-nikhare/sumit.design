MODULE · SAMASTA

Facial recognition for search and biometric enrollment

A consent-led facial scan flow used at PHCs to speed up beneficiary lookup, reduce duplicate creation, and optionally enroll facial biometric for future recognition. Designed with clear fallbacks: manual search, online search, and registration when no match is found.

Search-first Consent-led Online fallback OCR assist RBAC-aware

Screens and flow visuals below are exported from the module to document the end-to-end journey.

Problem

Fast identification without creating identity debt

In PHC operations, identity retrieval is a throughput problem. If staff can’t find a person quickly, they create a new record. That’s how duplicates become “normal”. Facial recognition is introduced as an optional accelerator, but it must remain safe, consent-led, and recoverable under network constraints.

What makes it hard

  • Consent and sensitivity: biometric capture cannot be “just another step”.
  • Match confidence: results may include multiple people with similar demographics.
  • Operational variability: lighting and background quality affect capture reliability.
  • Connectivity: online search is useful but cannot become a blocker for care.
  • RBAC reality: not every role should be allowed to enroll or edit biometric identity.

Design goal: keep facial recognition as a fast path with explicit consent and clear fallbacks.

Users

Same flow, different accountability

The flow is used inside facility contexts where speed matters, but governance still applies. Roles differ in what they can do: search, start OPD, enroll biometric, and link ABHA.

Front-desk / Facility staff

Primary: quickly find the right person and route them into the correct workflow (OPD / follow-up).

UX need: fast search, clear result cues, minimal rework.

MO / Authorized staff

Secondary: approve or perform sensitive actions like biometric enrollment where policy requires it.

UX need: explicit consent, auditability, and safe defaults.

Flow

Facial scan → results → online fallback → register if needed

The PDF shows three primary lanes: (1) face scan search success, (2) face scan search fail with online search, and (3) register face (biometric enrollment) with consent and capture guidance.

Step 01

Entry

“Search or Add People” from home.

Step 02

Choose mode

Face scan or search by mobile/name/ABHA.

Step 03

Consent + capture

Consent form → capture with guidance.

Step 04

Resolve

Select match → purpose of visit, or register new.

Key branches (as shown)

  • Scan success: show a results list (multiple candidates possible) and proceed after selection into “Purpose of Visit”.
  • Scan fail: show “No People Found” and offer “Search Online” without forcing loops.
  • Register face: enroll facial biometric with consent, capture guidance, image captured, and then return to the workflow.
  • If not found anywhere: create new person and optionally use OCR scan to prefill fields before saving.

Design decisions

Speed with safety: make the fast path trustworthy

1) Multi-mode search on one surface

“Search Facial Scan”, “Search People”, and “Add New People” live together so users don’t bounce between modules. Manual search remains first-class (mobile/name/ABHA), and face scan is an accelerator, not the only way.

2) Consent before capture, not after

The flow shows a dedicated consent form and clear “Don’t Add” / “Give Consent” options before any biometric action. This is deliberate: consent is a decision point, not a compliance checkbox.

3) Capture guidance as live feedback

The capture screens call out quality constraints (lighting/background, single-face, position in circle) so failure becomes actionable. Users get reasons like “Background not good” instead of mystery errors.

4) Online search is an explicit fallback

When local scan fails, the UI offers “Search Online” and shows “Searching online...” states, making the connectivity boundary visible.

5) OCR-assisted registration to reduce typing

When no match is found, the flow supports OCR scan and prefilled demographic fields to speed up creation and reduce entry errors.

Outcome: the module reduces duplicate creation pressure while keeping a safe path when biometric search fails.

RBAC as UX

Biometrics require tighter action boundaries than search

The design separates “find someone” from “enroll biometric”. Searching can be broader, but enrollment should be restricted to authorized roles and auditable contexts, because it changes future identification behavior.

Boundary 01

Search visibility

Users can search and view limited demographic cues to verify a match.

Boundary 02

Enroll / “Add Facial Biometric”

Enrollment is gated through consent and should be permission-controlled.

Boundary 03

Auditability

Biometric actions should produce logs so responsibility survives handoffs.

Edge cases

Designed for daily failure modes, not perfect demos

Poor lighting / cluttered background

Capture screens provide real-time cues (“Light good”, “Background not good”) and a how-to checklist.

No match found

User can switch to “Try Online Search” or proceed with new registration without being stuck in retry loops.

Consent declined

The flow supports “Don’t Add” paths so service delivery can continue without biometric enrollment.

Multiple results returned

Result cards include demographic cues (age, gender, village, masked mobile) to support correct selection.

Screens

Key surfaces in the module

These key screens show how facial recognition works as an accelerator, not a dependency. The flow makes consent explicit, guides capture quality in real conditions, and always provides a safe fallback to manual/online search and registration when no reliable match is found.

System fit

How facial recognition connects across Samasta

  • Beneficiary registration: used as a search accelerator and a de-dup support mechanism.
  • OPD services: reduces time-to-start by taking users straight into “Purpose of Visit” after selection.
  • ABHA linking: surfaced as “Link ABHA” when not linked/created, without blocking identification.

This module makes the first 30 seconds of the facility journey less chaotic. Which is the best you can hope for.

Impact

Lower friction at entry, fewer accidental duplicates

  • Improves beneficiary retrieval speed in high-throughput contexts.
  • Reduces duplicate record creation pressure by providing a fast recognition path with online fallback.
  • Makes consent and biometric capture explicit, improving trust and governance alignment.
  • Creates a cleaner handoff into OPD and follow-up workflows.

What’s next

Make exceptions manageable and enrollment more accountable

Exception queues

Surface “scan failed / online search unavailable / partial match” as reviewable states, not silent dead ends.

Stronger match verification

Improve match cues and confirmation steps to reduce wrong selection when multiple candidates appear.

Enrollment governance

Tighten role-action rules and audit logs around biometric enrollment so accountability is explicit.

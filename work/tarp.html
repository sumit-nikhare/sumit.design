TARP Design System

A unified component library built at Permute Solutions to speed up delivery and keep multiple products visually consistent. The system uses a grid-first layout approach, reusable components, and implementation-ready specs so teams can assemble complex workflows without redesigning the basics every sprint.

Project overview

A shared UI foundation for teams building multiple products in parallel

The TARP component library was created at Permute Solutions to support fast, consistent product delivery. Multiple applications were being built with different UI patterns, which meant teams were recreating the same components repeatedly and users were experiencing visual drift across modules.

What the design system had to solve

  • Reusable UI building blocks: a component library teams could ship with, not redesign every time.
  • Consistent visual language: typography, colors, iconography, and interaction rules aligned to the product brand.
  • Responsive patterns: components designed to scale across web and different breakpoints.
  • Implementation clarity: specs and documentation that reduce interpretation and rework.
  • Faster delivery: reduce build time by standardizing patterns across workflows like CRM, campaigns, analytics, and help centers.

Users

Designed for product teams first, so end-user UX stays consistent

The primary users of a design system are the people assembling product UI under time pressure. The system was structured to help designers and developers move faster while keeping application UI cohesive across modules.

Designers

Needed modular components, clear constraints, and consistent foundations (grid, spacing, typography) to design screens without reinventing patterns each time.

Developers

Needed reusable UI elements, documentation, and predictable variants so implementation stays consistent and doesn’t depend on ad-hoc decisions.

Stakeholders

Needed brand consistency across products, faster prototyping, and less design drift as teams built different modules in parallel.

End users (indirect)

Benefit from standardized patterns across workflows, which reduces relearning when moving between modules like CRM, campaigns, analytics, and knowledge base surfaces.

Approach

Grid-first foundations, then components, then applied screens

The system was built to support complex, data-heavy workflows. I anchored the library on a grid system so layout rules stayed predictable, then expanded into core components and patterns that could be reused across screens.

Role and responsibilities

  • Defined the design system architecture: components, patterns, layout rules, spacing, and interaction guidelines.
  • Built a modular component library in Figma for reusability and responsive behavior.
  • Established visual language (typography, color, iconography) aligned with the product brand.
  • Partnered with developers for implementation clarity through specs, documentation, and regular syncs.
  • Maintained the system through documentation and feedback loops for continuous improvement.

Design decisions

Decisions that kept the library scalable and implementation-friendly

1) Grid as a constraint, not a suggestion

Screens were assembled using a consistent grid system so layout doesn’t drift between teams. Components were designed to snap into predictable column structures.

2) Data-heavy components treated as first-class

Tables, metrics, lists, and filters were prioritized because they show up across CRM and analytics workflows and typically cause the most inconsistency when rebuilt repeatedly.

3) Variants designed up front

Components were built with predictable variants (density, states, and common combinations) to reduce one-off layouts and avoid “special case” UI.

4) Documentation as part of the deliverable

The system included usage guidance and implementation intent so developers could ship consistently without guessing how a component should behave.

5) Validate by building real screens

High-fidelity screens were used to verify that components held up in real workflows like knowledge centres, forms, and dashboards.

Governance

A design system only works if changes are controlled

To keep the system cohesive as more teams adopted it, the library needed clear ownership and a simple contribution loop. The goal was consistency without blocking delivery.

Boundary 01

Contribute

New components and variants follow shared layout rules and naming patterns.

Boundary 02

Review

Design rationale and implementation intent are reviewed before library updates.

Boundary 03

Adopt

Teams consume stable components and patterns instead of creating parallel UI.

Edge cases

Designed for density, long content, and real-world UI messiness

High-density data states

Tables and metric blocks were designed to stay readable with many columns, tags, and actions. Spacing and hierarchy were tuned for scan-ability.

Long labels and content overflow

Components account for long partner names, locations, and multi-value tags so UI doesn’t break when real data shows up.

Filter complexity without UI clutter

Filter patterns support multiple control types and clear reset behavior, while keeping the primary list state easy to scan.

Navigation that scales

Side navigation patterns support single and multi-level structures so products can grow without reworking information architecture every release.

System fit

A shared system that scales across CRM, campaigns, analytics, and support

The value of TARP isn’t a prettier UI library. It’s that common workflows can be built once and reused across applications without reinterpreting components every time.

Consistency across modules

Shared components reduce UI drift across modules like campaign management, CRM, FAQs, and analytics.

Faster prototyping and delivery

Pre-built UI patterns support rapid iteration while maintaining quality and predictable implementation.

Impact

Less reinvention, more consistency, fewer “but it looked different last week” bugs

  • Reduced duplicated component work by giving teams a shared library to build from.
  • Improved cross-product cohesion by standardizing foundations and interaction patterns.
  • Supported agile delivery by enabling faster prototyping and iteration cycles.
  • Validated system quality through applied screens across dashboards, forms, and knowledge surfaces.

MODULE · CHIP 2.0

Death notifications as a governed state change

A role-aware workflow to notify, validate, and confirm a death without turning it into a “one-tap delete.” The module treats death as a high-integrity system event: it is routed for confirmation, classified by age/sex logic, and only then permanently removes a beneficiary from programs.

Problem

“Mark as dead” is not a UI action, it’s governance

Death updates impact eligibility, program dashboards, and downstream reporting. If the UI treats it like a simple profile toggle, it creates two failures: accidental closures (data integrity) and slow, manual verification (operations). The module needed to support fast field notification without sacrificing confirmation controls.

What the Death module must solve

  • Governed completion: field staff notify; supervisors confirm or reject.
  • Predictable states: notified / pending / confirmed / rejected should be visible and auditable.
  • Right questions only: maternal and child cases require extra classification; others should stay lightweight.
  • Safe reversibility: cancellation and rejection flows must exist for inevitable errors.
  • Program-safe impact: confirmation must clearly communicate the consequences (removed from programs).

Users

One notifier, one confirmer: clean responsibility boundaries

The workflow is built around role responsibility, not shared access. Field users can notify quickly, while supervisory roles handle confirmation and rejections. This keeps the UI honest under RBAC and reduces the “who changed this?” ambiguity.

ASHA

Notifies a death from the member context, captures minimum required details (date, informant, place), and submits for confirmation. Can cancel a notification when an error is discovered.

ANM / CHO

Confirms or rejects notified cases. Applies additional classification where required (maternal/child), and controls the final irreversible state change.

Supervisory scale cues

The same flow supports different operating scales, so monitoring stays feasible without separate tools:

  • ASHA view: attention tile surfaces “Notified Cases” count for quick follow-up.
  • Village filters: notified cases grouped by village to reduce search friction.
  • Large registry context: the UI still works when the registry is in lakhs, not dozens.

Why this matters

Death isn’t just a record update. It alters program populations, removes people from eligibility pipelines, and changes household composition. So the UX has to respect operational authority.

Flow

Notify fast, confirm safely, classify only when needed

The module is designed as a small state machine: ASHA notifies, ANM/CHO confirms or rejects, and the system permanently marks the beneficiary as deceased only after confirmation. Classification is triggered by demographic logic, not by forcing every user through every form.

Step 01

Notify

Field user enters date of death, informant name, and place of death (home/facility/in transit).

Step 02

Route

Notification is routed for confirmation to supervisory roles instead of finalizing instantly.

Step 03

Confirm / Reject

Supervisor confirms and proceeds, or rejects with a reason when the notification is incorrect.

Step 04

Classify (if needed)

Maternal or child deaths trigger additional fields (cause, phase, symptoms) before final save.

Branching rules that keep the UX focused

  • Female 15–45: asks “Is this a maternal death?” and captures phase + probable cause when yes.
  • Child deaths: captures illness/injury history and symptoms, then probable cause.
  • Adults: captures probable cause with a lighter path, optionally including facility details when place is a facility.
  • Accidental deaths: requires selecting accident type (e.g., road accident, drowning, poisoning, suicide).

Design decisions

Designed to prevent irreversible mistakes without slowing the field

1) Minimum notify payload, maximum governance

ASHA captures only the essentials required to start governance (date, informant, place). The system explicitly communicates that the record will be sent for confirmation, reducing the “I finalized it” misunderstanding.

2) Confirmation is framed as irreversible

Before final confirmation, the UI warns that the beneficiary will be permanently marked deceased and removed from programs. This is not decorative copy. It’s a forced moment of accountability.

3) Classification is conditional, not universal

Maternal and child deaths require structured classification for reporting integrity. Everyone else gets a simpler path. That keeps the workflow usable while still supporting policy-critical data capture.

4) Rejection is explicit and reasoned

Rejecting a death notification isn’t a silent “no.” Supervisors must select a rejection reason (incorrect info, mistakenly notified, wrong information received). This creates feedback loops without requiring phone calls.

RBAC as UX

The interface enforces authority, not just the backend

The system prevents role leakage by making key actions appear only when the role owns the decision. This reduces clutter for field users and protects program integrity from accidental confirmations.

Boundary 01

Notify vs Confirm

ASHA can notify (and cancel their notification). Confirmation is reserved for ANM/CHO.

Boundary 02

Reasoned rejection

Supervisors reject with reason selection, keeping the system auditable and teachable.

Boundary 03

Program impact safety

Only a confirmed death triggers removal from programs, preventing premature eligibility changes.

Edge cases

Handled the messy stuff on purpose

Cancel notification (ASHA)

A dedicated cancel flow removes the beneficiary from the “Notified Cases” list with an explicit confirmation modal. This prevents wrong notifications from becoming long-running supervisory noise.

Constraint addressed: mistakes happen, and the UI shouldn’t pretend otherwise.

Reject with reason (ANM/CHO)

Rejecting a case requires selecting a reason (incorrect info / mistakenly notified / wrong information received), reducing repeat errors and lowering escalation calls.

Constraint addressed: feedback needs structure, not blame.

Place-of-death variability

Place of death supports facility, home, and in-transit scenarios. Facility routes optionally collect facility type/name without blocking the simpler cases.

Constraint addressed: field reality doesn’t fit neat dropdowns.

Accidental death classification

Accidental deaths route to a separate reason selection (e.g., road accident, drowning, poisoning, burn, suicide), making reporting consistent without forcing medical categories.

Constraint addressed: classification must be usable, not “perfect.”

System fit

Integrates with registry and program surfaces without breaking trust

Death updates touch multiple surfaces: family profiles, notified cases queues, and program eligibility. The module keeps those surfaces consistent by ensuring a single governed pathway controls the final “deceased” state.

Family context entry

The “Member not alive” action lives where field users already work, reducing navigation cost and ensuring the state change is grounded in identity.

Attention queues

“Notified Cases” appears as an attention item with counts, keeping follow-up visible without requiring daily hunting across registers.

Impact

Fewer wrong closures, clearer supervision, cleaner reporting

What this design improves

  • Reduced irreversible mistakes: notify vs confirm separation prevents one-tap finalization.
  • Faster supervisor processing: reasoned reject and conditional classification make review predictable.
  • Better auditability: confirmed state carries structured cause/phase where required.

Scale cues surfaced in the UI

Attention surface: Notified Cases count is shown prominently to drive follow-up.

Registry context: designed to operate when registered population is in lakhs, not small pilot datasets.

What’s next

Improve evidence and reduce repeat review loops

Evidence strengthening (without burden)

Add optional evidence fields that don’t block completion (e.g., free-text notes, facility reference context), while keeping the base notify payload lightweight.

Reason taxonomy consistency

Standardize rejection and accidental-death reasons into stable categories that support reporting while staying readable for frontline staff.

Faster supervisor review

Introduce a “quick review” path for high-volume days where supervisors can confirm/reject faster without opening deep navigation.

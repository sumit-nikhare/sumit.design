MODULE · CHIP 2.0

Profile that makes posting, facility, and scope explicit

Profile is the platform’s source-of-truth surface for who the user is, where they’re posted, and what facility and geography they operate within. It consolidates facility metadata, staffing reality (sanctioned vs posted vs vacant), SHC and village mappings, and ASHA area ownership so downstream modules route work correctly and governance isn’t hidden in admin backchannels.

Problem

At scale, wrong posting and weak mappings break every downstream workflow

In a multi-role public health platform, errors usually start with context. If a user is mapped to the wrong facility, if SHC and village associations are incomplete, or if ASHA areas don’t have clear ownership, the rest of the system behaves “correctly” but produces the wrong outcomes: misrouted work, missing lists, incorrect reporting, and constant supervisor escalation.

What Profile must solve at platform level

  • Make posting explicit: show where a user is officially posted and what that unlocks (and blocks).
  • Expose facility truth: consolidate facility metadata, staffing reality, and verification state in one place.
  • Reveal mapping gaps: surface incomplete SHC/village/ASHA area associations as visible states, not silent nulls.
  • Turn governance into work: approvals and verifications must be actionable, role-gated, and count-based.
  • Protect auditability: sensitive edits need controlled entry points and clear ownership boundaries.

Users

The same profile surface, different governance responsibilities

Profile serves everyone, but not equally. Frontline roles need clarity: where am I posted and what can I do today. Supervisory and admin roles need governance controls: approvals, verification states, and the ability to correct mappings without breaking audit integrity.

Medical Officer (PHC)

Needs an authoritative facility view: staffing availability, pending approvals, and SHC/village coverage. Uses Profile to resolve governance blockers before they cascade into missing work lists and incorrect reporting.

CHO / ANM (SHC)

Needs clean geography and ownership: which villages and ASHA areas fall under the SHC, which workers are linked, and what’s pending verification. Uses Profile to avoid wrong routing in registers and follow-ups.

ASHA

Primarily needs “my area” clarity: assigned coverage, linked facility chain, and who to contact for corrections. The UI is view-first with guided escalation, not edit-heavy.

Block / District admin

Uses Profile as a governance console: resolve approvals, validate facility and mapping changes, and keep administrative controls limited to what’s necessary for scale and auditability.

Flow

Confirm context → inspect mappings → act on governance

Profile is designed as a predictable inspection loop. Users start by confirming posting and scope, then move through facility slices (about, staff, SHCs, villages, ASHA areas) and finally resolve what’s blocking operations: approvals, verifications, or ownership gaps.

Behavior rules

  • Counts guide attention: totals vs pending states help users triage without scanning long lists.
  • Role-gated actions: edit and approval controls appear only when the user can legally act.
  • No silent gaps: unassigned areas and incomplete mappings are visible, named states.

Design decisions

Designed Profile as a governed system surface, not a settings page

1) Tabs for stable context

Facility information is sliced into predictable tabs (about, staff, SHC, villages, ASHA areas) so users can move across governance details without losing where they are.

2) Counts as operational anchors

Counts (total vs pending, linked vs unlinked) act as attention signals. This keeps the UI scannable on mobile and makes backlogs visible.

3) Staffing truth: sanctioned vs posted

The module separates official staffing structure from actual posting. Vacancy is treated as a real operating constraint, not hidden admin metadata.

4) Guided ownership assignment

Linking an ASHA area or assigning a worker is structured as a guided sequence with confirm steps to prevent accidental coverage changes.

5) Destructive actions are protected

High-impact edits (deletes, reassignments) are constrained by confirmation patterns and role-gated entry points to support auditability.

RBAC as UX

Same surface, different controls: visibility and action boundaries are role-shaped

Profile is where RBAC becomes tangible. The UI reduces confusion by showing what’s editable, what’s view-only, and what needs escalation. Supervisory roles get approval controls and verification queues; frontline roles get clarity and guided next steps.

Boundary 01

Edit permissions

Only roles with authority see edit and assign controls. Others see read-only fields plus escalation paths.

Boundary 02

Entity visibility

SHCs, villages, and areas shown are scoped by posting and geography so users don’t see irrelevant or unsafe admin breadth.

Boundary 03

Approval controls

Approval and verification states surface as counts and are routable actions for supervisory roles, not buried admin pages.

Edge cases

Built for real conditions: vacancies, unassigned areas, and partial mappings

Posting mismatch

When users are mapped to the wrong facility, the UI must make it obvious and provide a governed correction route instead of letting the error leak into every module.

Vacancy is operational truth

Sanctioned vs posted makes capacity constraints visible so missing follow-ups aren’t framed as user failure when staffing reality is the limiter.

Unassigned ASHA areas

Areas without ownership are surfaced as explicit states. Assignment and replacement flows are guided and confirm-based to prevent accidental coverage changes.

Partial mapping

SHCs can exist without complete village linking. The UI stays navigable while making “not verified / incomplete” visible so governance work is trackable.

Screens

These screens illustrate OPD as a routed, multi-role encounter instead of a single clinician form. Each surface is anchored to a “work due” state (vitals, consultation, labs, dispensing) so handoffs are explicit, resumable, and permission-safe across Nurse, MO, Lab Technician, and Pharmacist.

System fit

Profile powers routing, ownership, and reporting integrity across modules

Profile is upstream of everything. The facility and geography chain determines what cases appear, who owns follow-ups, which lists are in-scope, and how dashboards aggregate outcomes. Making this context inspectable reduces silent failures across the platform.

Routing & task ownership

Village and ASHA area ownership shapes what appears in registers, follow-ups, and case lists. Clean mappings reduce wrong assignments and missing beneficiaries.

Governance & reporting

Verification and approval states prevent polluted aggregation. Supervisory roles can resolve backlogs before they distort program dashboards.

Impact

Making context visible reduces support loops and prevents downstream “missing data” failures

  • Posting and scope clarity reduces misrouted work and incorrect facility reporting.
  • Visible mapping states make governance backlogs trackable instead of invisible blockers.
  • Sanctioned vs posted makes capacity constraints explicit for operational planning.
  • Role-gated edits protect auditability while keeping corrections possible at scale.

What’s next

What’s next: faster corrections without expanding permission scope

Correction requests with routing

Let frontline roles raise structured correction requests (posting, mapping, ownership) that route to the right authority, instead of informal calls and WhatsApp chains.

Bulk approvals with safeguards

For supervisory roles, add batch approval tools with clear filters and confirmation steps to reduce backlog without increasing error risk.

Lightweight change cues

When facility mappings or assignments change, show small “what changed” cues so teams don’t re-verify the world through phone calls.

MODULE · CHIP 2.0

Family Planning that behaves like governance, not a form

A role-aware Family Planning module built around eligible couple tracking, due follow-ups, and event-based status changes (notified → confirmed → closed). It supports frontline realities: low attention, interruptions, mixed connectivity, and strict program rules.

Problem

Family Planning fails when “eligibility” is invisible in the UI

In program delivery, Family Planning isn’t just counselling. It’s a governed lifecycle: who is eligible, what method they’re using, what’s due next, what was notified in the field, and what has been confirmed by the supervisor chain. If the UI doesn’t make these states explicit, the result is duplicates, wrong closure, missed follow-ups, and endless support dependency.

What this module must solve

  • Due-work first: follow-ups should surface as queues with dates, not buried inside profiles.
  • State integrity: notified vs confirmed vs closed must be visible and explainable to users.
  • Method follow-up that’s fast: structured questions, minimal typing, predictable next steps.
  • Governance handoff: ASHA can notify; ANM/CHO confirms; MO sees aggregates and exceptions.
  • Data completeness without drama: IDs, bank details, and history need clear edit ownership and prompts.

Users

The same couple, different responsibilities

Roles aren’t “views”, they’re accountability boundaries. The module treats each role as a different operator in the same lifecycle.

ASHA

Identifies events in the field (pregnancy, sterilization, death, marital status changes, infertility indicators) and submits notified updates with required context.

ANM / CHO

Confirms or rejects notified updates, maintains official status, and drives follow-ups. This role protects program integrity and prevents accidental closure.

MO / Supervisors

Consumes aggregates (coverage, no-method reasons, method mix), exceptions (pending certificates, high-risk), and geography-filtered lists to plan action.

System as a “program clerk”

The UI enforces what counts as “complete”: due dates, confirmation states, certificate status, and “not eligible” closures are treated as first-class UX states.

Flow

A lifecycle driven by due-work, events, and confirmation

The module is structured around three operational loops: (1) follow-up due execution, (2) status-change notifications, and (3) verification and closure. Users don’t hunt for work. Work finds them, with just enough context to act.

Loop 01

Due Follow-ups

Method follow-up, EC survey pending, routine follow-up queues.

Loop 02

Notified Events

ASHA submits changes with mandatory context and routing.

Loop 03

Confirm or Reject

ANM/CHO validates, prevents bad closure, preserves integrity.

Loop 04

Close + Track

Certificate states, inactive/closed cohorts, and follow-up scheduling.

What “method follow-up” actually does

  • Confirms current method: yes/no branching keeps the flow short for stable users.
  • Captures LMP + method details: injection date / patch date / condom quantity / IUCD expiry warnings.
  • Surfaces care + precautions: pre-authored guidance reduces variation and training load.
  • Collects issues + advice: a constrained set of symptoms with “add more” avoids free-text chaos.
  • Records fertility intent: “Do you want child/children” and timeline informs counselling path.

Design decisions

Made “status” usable, not just stored

1) Due-work is a primary surface

Follow-ups and pending survey work appear as counts and queues, so users can start execution without navigating into profiles. This matters because “time-to-first-action” is the real bottleneck in field work.

2) Notified cases are designed like a verification inbox

A notified event is not truth. It’s a claim that must be confirmed. The UI makes that explicit with clear states, reject/confirm actions, and “cancel notification” guardrails to prevent accidental program closure.

3) Filters are operational, not decorative

Category + status + geography filters allow supervisors to locate actionable subsets quickly (e.g., “Death cases in this village” or “Referred cases for sterilization”). The module treats filtering as a core task, not a secondary feature.

4) Profile composition supports partial completeness

IDs and bank details use explicit “Edit & Update” prompts and per-person completion indicators (updated vs not updated). This prevents silent missing data, while keeping the flow non-blocking for urgent work.

5) Counselling is structured to reduce variation

Counselling screens provide pre-authored, situation-specific guidance (newly married, one child, multiple pregnancies), and then ask for a decision. This keeps counselling consistent across workers and reduces training dependency.

RBAC as UX

Permissions define what becomes “official”

The module treats RBAC as the shape of the experience: who can notify, who can confirm, who can close, and what the system must show to prevent mistaken actions. The outcome is fewer “backend-only permissions” surprises and less UI clutter for frontline users.

Boundary 01

Notify vs Confirm

ASHA submits events; ANM/CHO confirms to make them official. UI messaging reinforces the handoff.

Boundary 02

Eligibility control

Closure states (e.g., sterilization, death, infertility flags) are gated to prevent accidental loss of service eligibility.

Boundary 03

Audit-friendly states

The UI preserves “who updated” and dates to make state changes explainable during review.

Edge cases

Handled the messy stuff on purpose

These aren’t rare. They happen daily. The system treats them as predictable UI states so users don’t invent workarounds.

IUCD expiry warnings

Expired IUCD is treated as a high-salience state with clear counselling instructions, so follow-ups don’t become “silent risk”.

Constraint addressed: safety messaging + low training bandwidth.

Cancel / Reject to prevent bad closure

Confirmation screens explicitly warn that rejecting/cancelling affects eligibility. This prevents accidental “removal from FP services”.

Constraint addressed: governance risk + supervisor-dependent workflows.

Exit without completing follow-up

Exit confirmations reduce drop-offs during multi-step flows, which are common during household visits and interruptions.

Constraint addressed: fatigue + interruptions + shared devices.

Multi-spouse / linked identities

Inactive cases and spouse-linked counts are handled as explainable lists instead of collapsing into one ambiguous record.

Constraint addressed: data integrity + real-world household complexity.

Screens

Key surfaces in the Onboarding module

These screens show onboarding as a governance flow: identity verification, posting scope selection, and a visible approval state before access. The UI makes “pending approval” a first-class experience, prevents duplicate registrations, and routes approved users directly into day-start so work begins without hunting through navigation.

System fit

Connects household registers to program execution

Family Planning doesn’t live in isolation. It sits on top of household registration and feeds into field planning. The module supports that by aligning its navigation and filters with the same geography structure (state → district → block → facility → village/ward), so work can be planned and supervised without cross-tool gymnastics.

Program-level insights

Coverage and method-mix summaries help supervisors spot where counselling or supply-chain follow-ups are needed.

Field execution surfaces

Today’s plan, due-work lists, and village-scoped views keep the module usable in the “few minutes per household” reality.

Impact

Designed for measurable program control

What the system can surface at a glance

  • Eligible couple coverage: 1,354 eligible couples, with 1,242 using a method (92%) and 112 on “No Method” (8%).
  • Execution load: follow-up due and survey pending counts are visible before users enter any profile.
  • Reason clarity for “No Method”: reasons like “Want child/children”, “Dislikes available method”, and “Other” can be summarized for targeted counselling.
  • Supervision control: notified cases, referred cases, and certificate-pending inactive cases can be filtered by geography and category.
  • Operational reach: dashboards can scale up to population-level views (e.g., registered population and attention-required counts), without changing the module’s mental model.

What’s next

Tighten state clarity where field reality hits hardest

Exception-first routing

Add explicit exception states to reduce manual follow-ups: “certificate pending”, “expired method”, “incomplete notification”, “missing ID”. These should appear as actionable chips and queue filters.

Audit-ready timelines in profiles

Expand follow-up history to show state transitions (notified → confirmed → closed) with “why” and “by whom”, so disputes don’t become phone calls and guesswork.

Make counselling outcomes trackable

Capture counselling outcome as structured values (accepted method, deferred, refused, referred) so program teams can measure conversion and not just activity.

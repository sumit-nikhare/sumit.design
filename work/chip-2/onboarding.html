CASE STUDY · CHIP 2.0

Onboarding that respects roles, approvals, and reality

A role-aware onboarding and day-start experience designed for multi-program operations, where identity, posting, and permissions are governance constraints, not profile preferences.

Mobile-first RBAC-driven UX Approval states Posting & facility mapping

*Full deck is in progress. This page captures shipped UX + system decisions.

Scope

What the onboarding system covers

Entry paths

Two ways in, depending on whether governance already exists.

Credentials login

Fast path for already-approved users, optimized for repeat daily usage.

  • Design intent: reduce friction for operational users who return every day.
  • Enables: quick access to role-scoped work without re-onboarding.
  • Prevents: accidental re-registration and support loops.

Outcome: faster entry with fewer retries and less confusion between “login” and “register”.

Mobile + OTP registration

Mobile-first identity capture for onboarding at scale when email or institutional IDs aren’t reliable.

  • Design intent: make identity entry possible on shared/low-end devices and inconsistent networks.
  • Enables: controlled creation of a new account before role and scope are applied.
  • Prevents: ghost accounts, mismatched identities, and partial sign-ups.

Note: OTP is identity entry before governance applies.

Governance handoff

Registration creates identity, posting, and approval state in one controlled chain.

Role + posting + facility mapping

Captures role and posting context as governance inputs, not profile decoration.

  • Design intent: establish scope and supervisory chain at the moment identity is created.
  • Enables: role-scoped navigation, assignment eligibility, and data visibility boundaries (district/block/facility).
  • Prevents: incorrect access, cross-facility data leakage, and admin clean-up work.

Result: permissions shape what users see, not what they fail at later.

Approval states

Registration does not equal access. Pending / approved / duplicate are explicit UX states.

  • Design intent: make “waiting” explainable instead of looking like failure.
  • Enables: request traceability (ID + details), controlled activation, accountable approvals.
  • Prevents: duplicate registrations, repeated OTP attempts, and “app is broken” perception.

Result: fewer retries and cleaner identity records.

Day-start surface

Post-approval landing prioritizes execution: punch-in establishes daily state, work plan answers “what next”.

  • Design intent: shift users from navigation to action within the first screen.
  • Enables: consistent day-start flow across modules without exposing everything at once.
  • Prevents: menu wandering, missed activities, and inconsistent task completion.

Result: faster time-to-first-task and higher day compliance.

Flow

From “unknown” to “operational”

The flow is intentionally linear: users either authenticate or register, then move through approval before gaining access to role-scoped features. No permission surprises, no dead ends.

Design decisions

Built to scale across roles and programs

These decisions keep onboarding scalable across programs, geographies, and partner organizations without turning the UI into a permission minefield. I established role, scope, and approval state early so governance is enforced through predictable UI states instead of late-stage access errors.

1) Split entry paths early

I separated authentication (approved users) from onboarding (new users) so each path stays fast, predictable, and explainable. Combining them creates ambiguity and increases repeat attempts.

  • Users see only the path relevant to their account state
  • Reduces accidental re-registration and OTP retry loops

2) Treat posting as governance

Posting and facility mapping aren’t profile fields. They define data boundaries, approval routing, and assignment eligibility. I designed posting selection as a governance step so scope is correct before users enter operational surfaces.

  • Prevents cross-facility visibility and wrong-task eligibility
  • Reduces reassignment and “fix my posting” escalations

3) Make approval states visible

Approval is a legitimate system state, not a backend delay. I surfaced pending/approved/duplicate as explicit UI states with traceability so users understand what’s happening and what to do next.

  • Waiting becomes explainable and trackable
  • Fewer repeat submissions and support escalations

4) Day-start over navigation

After approval, users need execution, not exploration. I prioritized day-start actions (punch-in + work plan) so the first screen answers “what now” based on role and daily state.

  • Fewer irrelevant actions and less menu wandering
  • Consistent entry into work across modules

RBAC as UX

Permissions influence what users see, not what they fail at

Instead of showing everything and letting permissions break later, onboarding sets role + scope early so navigation and actions are scoped from the first operational screen.

Signal 01

Role

Determines module visibility and action-level permissions (create/edit/approve/assign).

Signal 02

Scope

Posting + facility mapping sets data boundaries and drill depth.

Signal 03

State

Approval + punch-in states gate actions and keep the experience explainable.

Edge cases

Handled the messy stuff on purpose

These are not “rare.” They’re daily behavior in field conditions. Each edge case is handled as a predictable UI state to reduce retries and support dependency.

*High-frequency failure modes treated as first-class states, not exceptions.

OTP

Weak network / resend loops

Resend timer + explicit affordance prevents panic tapping and repeated OTP requests, especially in low-connectivity zones where delivery is delayed and users retry by instinct.

Constraint addressed: intermittent connectivity + shared devices + low tolerance for waiting. System effect: reduces OTP flooding, verification churn, and “stuck onboarding” support calls.

Identity

Mobile already registered

Clear messaging + recovery/escalation path routes users to account recovery or supervisor help instead of creating duplicates that split identity and records across facilities.

Constraint addressed: data integrity + governance ownership for account activation. System effect: prevents duplicate identities, conflicting posting scopes, and reconciliation work.

Wizard

Exit mid-registration

Exit confirmation prevents accidental abandonment of multi-step onboarding during interruptions (calls, travel, network drops), treating interruption as expected behavior.

Constraint addressed: fatigue + interruptions + form length on mobile. System effect: reduces partial submissions, repeated starts, and supervisor corrections.

Approval

Pending approval

Request ID + “view details” makes waiting explainable and traceable, letting users verify what they submitted and understand approval is the gate.

Constraint addressed: supervisor-dependent workflows + asynchronous activation. System effect: reduces repeat submissions, duplicate requests, and workaround behavior.

MODULE · CHIP 2.0

Onboarding that treats identity + posting as governance

A role-aware login and registration flow that establishes identity, posting location, and approval state upfront. The UI prevents duplicate registrations, makes “pending approval” a first-class state, and routes approved users into a day-start surface so work begins without hunting through menus.

Problem

Onboarding breaks when identity and approvals are treated as backend-only logic

CHIP 2.0 spans programs, facilities, and role hierarchies. If onboarding is a generic “create account” flow, the platform inherits messy identity data, wrong posting scope, and invisible approval states. In the field, that becomes repeat attempts, duplicate users, and support dependency before anyone can do actual work.

What onboarding must solve at scale

  • Split entry paths: Existing users should authenticate quickly; new users should register with OTP and guided steps.
  • Posting as scope: District/block/facility mapping defines what modules and records a user can access.
  • Approval as a UX state: “Pending approval” must be actionable and explainable, not a dead end.
  • Duplicate prevention: Mobile already registered should route to supervisor escalation, not create split identities.
  • Day-start readiness: Once approved, users need a clear first task surface (punch-in + work plan) to begin execution.

Users

Two user modes: approved staff logging in vs field users registering in real conditions

The onboarding system supports both credentialed staff who are already provisioned and frontline workers who register via mobile OTP. It also accounts for higher authorities who can correct details before approval, and supervisors who act as the escalation path when identity conflicts occur.

Already-approved users

Login with username/password or direct credential flow. Goal: get them into day-start with minimum friction.

New field registrants (OTP)

Mobile-first registration with short steps for personal + employment details and posting selection.

Supervisors / higher authorities

Review requests, correct role/posting/facility mapping, and approve. Their edits must be transparent to the registrant.

Program admins

Own policy-driven roles, module visibility, and approval routing rules across districts, blocks, and facilities.

Flow

A split-path flow that resolves identity, scope, and state before access

The flow is designed as two predictable paths: credentials login for provisioned users, and OTP-based registration for new users. Both converge on the same governance states (pending approval / approved), and approved users land on a day-start surface that gates work until punch-in.

Step 01

Choose entry path

Login for existing users, Register for OTP onboarding.

Step 02

Capture identity + posting

Personal + employment details, district/block/facility mapping.

Step 03

Approval state

Pending request with Request ID, details view, and supervisor contact.

Step 04

Day-start

Punch-in gate + work plan to start execution without navigation.

Routing behavior

  • Credentials users skip registration and authenticate directly.
  • Registration creates a reviewable request and surfaces status with a clear next step.
  • Approved state routes into day-start and exposes module entry points based on role + posting.

Design decisions

Designed onboarding as governance-first UX, not a generic sign-up

1) Split entry paths early

Login and registration are not the same job. Separating them reduces confusion, prevents retries, and keeps approved users fast.

2) Treat posting and facility mapping as governance

District/block/facility selection determines scope, supervisory chain, and module visibility. The UI treats it like a critical setup step, not optional metadata.

3) Make approval states visible and explainable

“Pending approval” is presented as a valid state with a Request ID and a details view so users don’t assume the app is broken.

4) Build explicit anti-duplication messaging

When a mobile number is already registered, the UI blocks creation and routes to supervisor resolution to avoid split identities.

5) Route approved users into day-start, not navigation

Punch-in and a work plan answer “what do I do now” immediately. This reduces menu hunting and aligns with daily execution rhythms.

RBAC as UX

RBAC starts at onboarding: scope and actions are established before users enter modules

Onboarding sets the platform’s RBAC baseline by binding a user to a role, posting location, and facility. That mapping determines what modules appear, what records are accessible, and which actions are available. Approval is the governance checkpoint that prevents incorrect access from entering the system.

Boundary 01

Module visibility

Home surfaces show only modules relevant to the user’s role and posting, reducing clutter for frontline staff.

Boundary 02

Approval routing

Requests route to the correct supervisory chain; edits before approval are transparent via a details view.

Boundary 03

Day-start gating

Punch-in and daily state ensure users don’t start tasks without context, assignments, or work plan entry.

Edge cases

Handled the messy stuff as predictable UI states

Weak network / resend loops

OTP screen includes a resend timer and a clear resend affordance to prevent panic tapping and repeated requests.

Mobile already registered

Blocks duplicate registration and routes users to supervisor support, protecting data integrity.

Exit mid-registration

Exit confirmation prevents accidental abandonment of multi-step registration flows.

Pending approval waiting

Request ID + “View details” makes the waiting state legible and reduces repeated submissions.

Screens

Key surfaces in the Onboarding module

These screens show onboarding as a governance flow: identity verification, posting scope selection, and a visible approval state before access. The UI makes “pending approval” a first-class experience, prevents duplicate registrations, and routes approved users directly into day-start so work begins without hunting through navigation.

System fit

Onboarding connects governance setup to daily execution

Once approved, onboarding hands users into a day-start surface that makes daily state visible (punched-in vs not), surfaces assigned activities, and provides direct entry into registers. This prevents the common failure mode where users are “logged in” but still don’t know what to do next.

Day-start as the default landing

Punch-in is treated as a work gate. When a user hasn’t punched in, the UI explains why work is blocked and offers a single action to proceed.

Work plan + register entry points

Activities and register counts provide a clear starting point for execution and reduce reliance on supervisors for “what now.”

Impact

Fewer retries, fewer duplicates, and a clearer first day experience

  • Reduced confusion by separating login vs registration paths and converging on clear approval states.
  • Prevented duplicate identities by blocking mobile re-registration and routing escalation to supervisors.
  • Made governance visible with Request ID + details view, lowering “app is broken” perception.
  • Improved time-to-first-task by routing approved users into punch-in + work plan instead of menus.

In-product counters surfaced on day-start (example): Registered population 12,54,872; Attention required 1,462; Family registered 21; Family DHS completed 18; People registered 75; People DHS completed 62; ABHA linked 21; ABHA created 18.

What’s next

What’s next: tighten onboarding integrity without adding steps

Proactive duplicate resolution

Offer a guided supervisor resolution flow (who to contact, what info to share) so “already registered” doesn’t become a dead end.

Offline-first OTP recovery

Make resend behavior resilient under low connectivity with clearer retry states and safe fallbacks.

Approval transparency for supervisors

Add lightweight audit cues (what changed, by whom) to reduce back-and-forth and speed up approvals.

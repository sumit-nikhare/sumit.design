MODULE · CHIP 2.0

VHSNC Meetings as a governed execution workflow

A village-level planning, conduct, and verification system where “meeting happened” is treated as a trackable state machine. The UI supports role-based responsibility (ASHA executes, supervisors verify), evidence capture, and scalable rollups across SHC and PHC geographies.

Plan → Conduct → Verify Evidence-first Village rollups Geo filters + FY RBAC as UX

*Derived from the VHSNC Meeting planning, conduct, cancellation, and verification screen set. Replace placeholder screenshots with final exports when ready.

Problem

Village meetings fail quietly when the system can’t prove execution

VHSNC meetings are operationally messy by default: schedules change, assignees vary, connectivity is uneven, and “completed” often means “someone said so.” Without explicit states and evidence capture, supervisors are forced into manual follow-ups, and reporting becomes a trust issue.

What the module must solve

  • Governed completion: “Conducted” should be separate from “Verified.”
  • Predictable states: upcoming/due/overdue must drive action without calls.
  • Evidence capture: attendees + photos + register image make audit possible.
  • Scale rollups: supervisors need village counts and filters, not a long list of screens.
  • Edge-case handling: cancellations, deadline-crossed meetings, and re-planning must be first-class flows.

Users

Three roles, three intents: execute, govern, monitor

The module is designed around responsibility boundaries. Field execution is delegated, while governance and oversight live with supervisory roles across SHC and PHC levels.

ASHA

Executes the meeting flow when assigned: conducts the meeting, captures evidence, and submits for verification within the allowed time window.

ANM / CHO (SHC)

Plans meetings for villages, assigns ASHA responsibility, manages edits/cancellations, and verifies submissions when acting as the supervisory verifier.

MO (PHC)

Monitors execution at scale using rollups, filters, and village profiles. Verifies or rejects pending submissions to enforce reporting integrity.

Scale snapshots

The same module supports different workload realities:

  • ANM view: Total villages 10 (Nov), showing planned/due/conducted/not-held rollups.
  • ASHA view: Typically 1 village per month for execution and submission.
  • MO view: Total villages 42 (Nov), enabling PHC-level monitoring and filters.

Flow

A state machine that makes governance visible

VHSNC is modeled as a monthly meeting per village with explicit state transitions. The goal is not “more UI,” it’s fewer ambiguous outcomes.

Step 01

Plan

Select month, date, meeting place, and assign ASHA.

Step 02

Execute

Attendees selection + group photo + register photo + notes.

Step 03

Submit

Meeting moves to Verification Pending with audit context.

Step 04

Verify

Supervisor verifies or rejects with reason.

Statuses that drive behavior

The module treats statuses as the primary navigation layer:

  • Planning: Not Planned → Planned
  • Time-based: Upcoming → Meeting Due → Overdue
  • Governance: Verification Pending → Conducted & Verified or Rejected
  • Exceptions: Cancelled, Not Held (deadline crossed)

Design decisions

Designed for accountability without punishing field users

1) Rollups first, lists second

Each role starts with monthly rollups by village count (planned/due/conducted/not-held), so supervision is about coverage, not scrolling. Lists are then filtered by the selected status.

2) Time cues are explicit and actionable

“Meeting upcoming in 4 days,” “Meeting due today,” and “X days left to update meeting” reduce ambiguity and replace informal reminders with predictable UI behavior.

3) Evidence capture is staged, not dumped

Conduct flow is split into: attendees selection → group photo → register photo → notes. This keeps field entry doable while ensuring the submission contains minimum audit proof.

4) Village profile supports continuity

Village meeting history across months makes exceptions visible (cancelled/not held/rejected), so follow-ups aren’t based on memory or chat threads.

RBAC as UX

Permissions are enforced through visible responsibility

The system makes governance unavoidable by separating who can do what. This reduces “fake done” meetings and keeps supervisors in control without blocking field execution.

Boundary 01

Assignee gating

“Conduct” and “Cancel” actions appear only for the assigned ASHA and only within allowed meeting windows.

Boundary 02

Verification control

Supervisors can verify or reject pending meetings, including selecting rejection reasons.

Boundary 03

Geo scope

SHC sees villages under the SHC; PHC sees villages across SHCs, using geography filters and month/FY controls.

Edge cases

Built around reality: cancellations, deadlines, and retries

Exit guardrails

Planning and conduct flows include explicit “exit without saving” confirmation to prevent silent drop-offs in multi-step capture.

Cancellation is irreversible

Cancelling requires a reason and blocks meeting conduct, avoiding “cancelled but still completed” data corruption.

Not Held as a valid end state

When the meeting deadline crosses, the system records Not Held with reason, instead of leaving villages stuck in “Overdue forever.”

Rejected is inspectable

Rejections don’t disappear. They remain in village history so teams can identify repeated failure patterns and coach execution.

Screens

Key surfaces in the Onboarding module

These screens show onboarding as a governance flow: identity verification, posting scope selection, and a visible approval state before access. The UI makes “pending approval” a first-class experience, prevents duplicate registrations, and routes approved users directly into day-start so work begins without hunting through navigation.

System fit

Works at village execution and PHC monitoring scale

The module supports two different truths at once: ASHA needs a single clean path to completion, while MO needs cross-village oversight. Filters bridge that gap without introducing separate products.

Geography filters

State → District → Block → PHC/CHC → SHC → Village/Ward enables supervisors to slice execution without manual coordination.

Time period controls

Financial Year + Month selection aligns reporting and reduces ambiguity when reviewing historical meeting states.

Impact

Reduced “trust-based reporting” by making verification the default

Observed outcomes from the design

  • Supervisor-ready rollups: village counts by status reduce follow-ups and make workload visible.
  • Higher data integrity: Conducted vs Verified separation prevents premature completion.
  • Fewer retries: clear time cues (“due today”, “days left”) reduce missed windows and rework.

Real scale cues from the screen set

ANM month example: 10 villages with rollups like 5 planned / 2 due / 2 conducted / 1 not held.

MO month example: 42 villages with 38 planned / 12 due / 22 conducted / 4 not held, plus verification pending queues.

What’s next

Tighten governance without adding friction

Exception clarity

Expand cancellation and not-held reasons into structured categories to support PHC-level pattern analysis and targeted coaching.

Verification speed

Add a lightweight “quick review” mode for supervisors where evidence can be reviewed and verified in fewer steps for high-volume months.

Better re-plan continuity

When meetings are cancelled or rejected, make “Re-plan” carry forward the context (place, assignee, previous reason) to reduce repeated entry.

MODULE · CHIP 2.0

Sickle Cell screening + IMR/MMR household capture, built as governed UI states

A field-ready set of DHS-adjacent flows that capture: (1) Sickle Cell screening status, DBS sample collection, barcode validation, and follow-up needs, and (2) household-level infant and maternal death event capture with structured, repeatable “individual details” steps. The design treats follow-up and data integrity as primary outcomes, not side effects.

Problem

Field capture fails when the UI hides follow-up and integrity constraints

These flows aren’t “forms”. They’re compliance-aware sequences with failure modes: lab reports may not exist, DBS kits may be unavailable, barcode scans can collide across beneficiaries, and household event data requires repeatable individual-detail entry without losing partial work.

What this module must solve

  • Branching that stays understandable: last 2 years → report received → outcome/card, without forcing impossible inputs.
  • DBS traceability: scan barcode, verify beneficiary, store collection date, and prevent reuse of DBS numbers.
  • Follow-up as a visible state: “lab report pending”, “DBS sample pending”, “kit unavailable”, “training required”.
  • Repeatable individual details: capture up to 5 entries for maternal/infant events without breaking navigation or losing work.
  • Interruptions: back/exit confirmation so accidental navigation doesn’t create silent data loss.

Users

Designed for frontline capture with supervised follow-up ownership

The primary user is a field worker capturing data under time pressure and real constraints. The system must also create clear follow-up obligations that can be acted upon by supervisory roles.

Field worker

Captures screening status, collects DBS samples, scans barcodes, and records household event details.

Supervisor / program team

Acts on follow-up queues: pending lab reports, DBS pending due to kit/training constraints, and verification needs.

Flow

Branch early, then enforce integrity at the point of capture

The overall structure is intentionally predictable: a short “status gate” determines the path, and each path has a clear completion rule (save + success confirmation). When capture cannot complete, the UI must convert it into a follow-up state.

Step 01

Status gate

“Test in last 2 years?” and “Report received?” define the path.

Step 02

Outcome + card

Outcome selection + card type captured when available.

Step 03

DBS collection

Yes/No → scan barcode → validate uniqueness and beneficiary.

Step 04

Household events

Maternal/infant event counts → up to 5 individual detail steps.

Routing behavior

  • Path 1: Test done + report received → capture outcome and card type → save.
  • Path 2: Test done + report not received → show “lab report pending” + follow-up cue → save.
  • Path 3: Test not done / don’t know → DBS collection instructions → collect/scan or record reason → save.

Design decisions

Made “can’t complete” outcomes explicit, actionable, and saved

1) Status questions first, details later

The first two questions (“last 2 years” and “lab report received”) prevent long scrolls and keep users out of irrelevant screens.

2) Lab report pending is treated as a valid saved state

If reports are not available, the UI shows “Lab Test Report is Pending” with a follow-up message and still allows Save. That prevents drop-offs and repeat entry.

3) DBS integrity is enforced at scan time

Barcode scanning includes duplicate detection (“DBS number already exists”) and supports re-scan. When a number is captured, the UI prompts verification that the number belongs to the correct beneficiary.

4) Non-collection reasons are structured

When sample isn’t collected, the UI captures why (beneficiary denied, not present, kit unavailable, training required) and creates a follow-up state instead of silently failing the flow.

5) Multi-entry event capture stays navigable

Maternal/infant event details support up to 5 entries with a clear stepper (Women 1–5 / Child 1–5), so users don’t lose context mid-way.

RBAC as UX

Authority is expressed through what can be finalized vs what becomes follow-up

This module assumes not everyone can resolve every dependency. The UI distinguishes capture from resolution: frontline users can record status and constraints; supervisory roles can address backlogs and unblock completion.

Boundary 01

Capture vs resolve

Frontline capture creates “pending” states that can be acted on later without re-entry.

Boundary 02

Traceability

DBS barcode uniqueness and beneficiary confirmation protect downstream data integrity.

Boundary 03

Controlled loss

Exit/back confirmation reduces accidental loss during multi-step entry.

Edge cases

Handled the messy stuff on purpose

Lab report not received

“Lab report pending” is shown as a stable message with a follow-up suggestion, while still allowing Save.

DBS number collisions

When a DBS number is already associated with another beneficiary, the UI blocks the Save path and forces re-scan, preventing silent duplicates.

No kit / no training / beneficiary not present

Non-collection reasons are captured as structured inputs that convert into “Follow-up needed” states.

Accidental navigation loss

Back confirmation (“data will be lost”) protects multi-step entry from accidental exits in mobile conditions.

System fit

Works because it’s designed as states inside a larger DHS ecosystem

These flows integrate naturally into a DHS-style household system: structured questions, repeatable individual entries, and a clear definition of “saved but pending” vs “completed”. The UI makes it obvious when work is complete and when it must return as follow-up.

Follow-up as a system output

“Lab report pending” and “DBS sample pending” become trackable obligations, not forgotten dead ends.

Data integrity at the source

Barcode collisions and beneficiary verification reduce downstream cleanup and reduce reliance on manual reconciliation.

Impact

Higher completion without hiding reality

  • Converted “can’t complete” situations into explicit saved states (pending report, DBS pending) instead of drop-offs.
  • Protected traceability using DBS scan + duplicate prevention + beneficiary confirmation.
  • Reduced repeat entry and support dependency through stable success messaging and predictable steps.
  • Enabled structured, multi-entry household event capture without turning it into a scroll nightmare.

UI-level specificity represented in the flow: 3-way status options (Yes/No/Don’t know), report received (Yes/No), outcome selection (Disease / Trait / None), card options (Red/Yellow/White/Not available), DBS capture (Yes/No), non-collection reasons (4 structured options), and event detail entry supporting up to 5 individuals.

What’s next

What’s next: make follow-up operational, not just visible

Turn pending states into a queue

Route “Lab report pending” and “DBS pending” into a role-scoped queue with due timing, so follow-up isn’t dependent on memory.

Add lightweight audit cues

Show “captured by / captured on” and last updated timestamps for multi-step household events, so reviews and corrections become easier.

Reduce scanning retries

Improve re-scan guidance with clearer instructions and a quick “manual entry fallback” that still enforces collision checks.

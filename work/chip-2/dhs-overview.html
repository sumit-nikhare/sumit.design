MODULE · CHIP 2.0

Mobile dashboard for ABHA + DHS progress, across admin levels

A drill-down “progress + attention” surface for supervisors to track registration and DHS completion from State → District → Block → PHC/CHC → SHC → Village/Area. It supports time windows (All time / Today / This week / This month), demographic splits, and verification queues so action is obvious without exporting reports.

Problem

Progress dashboards fail when they show numbers without operational next steps

Supervisors don’t just need “how much.” They need “where is it stuck” and “what action unlocks progress.” In multi-level systems, a state-level percent means nothing if villages are unverified, facilities are unmapped, or new users are pending approvals. This dashboard treats progress, verification, and geography as one system.

What mDash must solve

  • Single metric spine: total registered and % coverage against target, consistent across levels.
  • Drill-down by geography: state → district → block → facility → SHC → village/area.
  • Time windows: all time vs today/week/month with “change from previous” to show movement.
  • Quality layers: DHS completion, demographics, family indicators as “depth”, not separate reports.
  • Attention required: verification/approval counters that point to the governance backlog.

Users

Different roles, same truth, different depth

The dashboard supports supervisors who need prioritization and drill-down, while still remaining readable for facility owners. RBAC determines what level of geography a user can view and which “attention” queues they can resolve.

District / Block supervisors

Track coverage, identify low-performing SHCs/areas, and prioritize verification and approvals that unblock field execution.

Facility owners (PHC/CHC)

Monitor OPD and registration activity, see verification backlogs, and check whether the facility is moving or stalled.

Program leads

Use DHS completion and indicator splits to understand depth of data capture, not just raw registration volume.

Governance roles

Work through approvals and verification backlog: health facilities, villages/wards, and new user registrations.

Flow

Select time → select geography → see progress → act on backlog

The interaction model is intentionally repetitive: users pick a time window, pick a geography lens, and read the same “progress spine”. That consistency enables fast scanning without retraining across screens.

Step 01

Choose time

All time / Today / This week / This month with deltas vs previous.

Step 02

Choose geography

State → District → Block → Facility → SHC → Village/Area.

Step 03

Read progress

Total registered + % against target, plus distribution lists/map.

Step 04

Go deeper

Demographics, family indicators, DHS completion, and attention queues.

Routing behavior

  • Map vs list parity: districts can be seen as a map view or list view, keeping the same metric spine.
  • Consistent headers: the same “PHC/District/Village/State : Name” header prevents confusion about current scope.
  • Geo filter modal: state/district/block/PHC/SHC/village/area with clear apply/clear actions.

Design decisions

Designed for scan-speed, not report-reading

1) A single progress spine across every level

“Total registered people + % vs target” stays identical whether the user is at PHC, district, or state. Only the breakdown changes.

2) Time windows show movement, not just status

Today/Week/Month show registration totals with change vs previous period, so supervisors can detect momentum shifts early.

3) Drill-down lists are ranked and comparable

Sub-health centers and areas are listed with counts and % so the “lowest” becomes immediately visible without extra filters.

4) Depth metrics live next to volume metrics

DHS completion and demographic splits sit under the same dashboard surface, reducing the gap between “coverage” and “quality”.

5) “Attention required” is first-class

Verification/approvals are shown as counters (not buried) so governance work is treated as a blocker, not an afterthought.

RBAC as UX

Access is scoped by geography, and actions are scoped by authority

mDash is valuable only if it respects organizational scope. A village-level user should not browse state performance. A state admin should see verification backlogs and program insights. The UI assumes geography scoping is a primary constraint.

Boundary 01

Geography visibility

Users see only their permitted geo tree (state/district/block/PHC/SHC/village/area).

Boundary 02

Data depth

Program leads get DHS + indicator insights; facility owners focus on operational activity.

Boundary 03

Action authority

Only governance roles can resolve approvals/verifications surfaced as “attention required”.

Edge cases

Handled mixed deltas, partial coverage, and governance backlog visibility

Negative deltas

Weekly views can show negative change versus previous periods. The UI keeps deltas visible so backsliding isn’t hidden.

Uneven SHC coverage

SHCs show wide spread in coverage (examples range from 82% down to 49%), so lists must support fast prioritization.

Approval/verification backlog

“Verifications / approvals required” is treated as a headline operational blocker, not a buried admin view.

Map + list parity

Districts can be seen in map view or list view, using the same values so users don’t interpret two “truths”.

Screens

Key surfaces in the Onboarding module

These screens show onboarding as a governance flow: identity verification, posting scope selection, and a visible approval state before access. The UI makes “pending approval” a first-class experience, prevents duplicate registrations, and routes approved users directly into day-start so work begins without hunting through navigation.

System fit

mDash connects coverage to governance and execution

This dashboard is not a “nice chart.” It ties together coverage, data depth, and the governance backlog that blocks progress. It also sits alongside day-start and registers, so the system can move from insight → action without changing tools.

Coverage + depth

ABHA registration/linking totals show reach, while DHS completion and indicators show whether the data is usable for programs.

Governance backlog

Verification/approval counters keep facility/village/user governance visible as an operational dependency.

Impact

A dashboard that makes prioritization defensible

  • Made coverage scannable with a consistent “total + % vs target” spine across levels.
  • Enabled rapid triage with ranked geo lists (district/SHC/area performance side-by-side).
  • Connected volume to depth by surfacing DHS completion alongside registration totals.
  • Reduced “unknown blockers” by exposing verification/approval backlog as a first-class operational signal.

Example stats represented in the UI: Total registered people 6,54,872 (~83.5%) with target 7,83,872; ABHA link/created shown at 6,54,872 (89%); DHS completed people 3,54,872 (52%); weekly registered example 11,000 with +1,200 (20%) vs previous week; SHC coverage examples range from 82% down to 49%; attention required example counters: 08 health facilities, 12 villages/wards, 05 new user registrations.

What’s next

What’s next: reduce ambiguity, increase trust

Explain the target and denominator

Add lightweight “what target is this” help text at every level to avoid misinterpretation when targets change by program or geography.

Turn “attention required” into navigable queues

Clicking a counter should take users directly into the verification/approval workflow, scoped to the same geography.

Confidence cues for freshness and sync

Add “last synced” and partial data warnings so supervisors don’t punish teams for network reality.

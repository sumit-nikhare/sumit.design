MODULE · CHIP 2.0

Officer Directory: find the right person, at the right level, without calling 5 people first

A role-aware directory that maps District → Block → PHC → SHC → Village chains, and organizes contacts across health workers, elected members, and committee roles. The UX is designed for field operations: fast calling, clear ownership, and governance-aligned edits.

Problem

Operational work slows down when contact chains are informal

Field execution depends on escalation paths and accountability. But in most deployments, “who do I call?” lives in WhatsApp threads, personal phonebooks, and outdated lists. The result is predictable: missed follow-ups, duplicate outreach, and delayed escalation.

What this directory must solve

  • Trustworthy hierarchy: District/Block/PHC/SHC/Village needs consistent drilldown, not random lists.
  • Role grouping: Users need “Health Workers”, “GP Elected Members”, and “VHSNC/WHC Committee” as stable buckets.
  • Fast calling: Contact cards must prioritize name + designation + phone, with one-tap call.
  • Governance-grade edits: Adding/updating members must be structured and reviewable, not free-form chaos.
  • Facility readiness: Teams need quick access to nearest FRU / Blood Bank / Collection Unit with distances.

Users

Same directory, different “default scope” per role

The module behaves like a shared service, but entry points and defaults change by role. ASHA starts from catchment context, ANM from SHC context, MO from PHC context. Everyone still drills up and down the same ladder.

ASHA

Sees catchment summary (example shown: Registered Population 6,54,872 and a target), and uses “My Team” for escalation plus “Important Facilities” for urgent routing.

ANM / MO

Works from facility scope (PHC/SHC) and needs committee + elected member details for coordination, plus higher-level officers for approvals and escalation.

Flow

Drilldown first, then act: browse → select → call / update

The directory prioritizes stable navigation over “search everything”. Users select level (District/Block/PHC/SHC/Village), then view grouped lists and act from contact cards.

Design decisions

Built as a governance service, not just a contact list

1) Hierarchy is the primary IA

District/Block/PHC/SHC/Village tabs are treated as the core information architecture so users can predict where a person “belongs”. This matters in government systems because escalation depends on scope, not convenience.

2) Grouping reflects how work actually happens

Lists are organized into Health Workers, Elected Members, and VHSNC/WHC Committee, because coordination needs different sets of people at different times, and mixing them increases time-to-call.

3) “Add/select member” is structured, not free-form

Add flows use designation pickers (e.g., Village President, Panchayat Development Officer, School Headmaster, ASHA/AWW, CHO/PHCO), then capture name + phone, then confirm (“Please check all details before saving”). This reduces duplicates and prevents garbage data from becoming the official directory.

4) Committee account details are treated as a governed task

VHSNC/WHC “Account Details” includes bank name, branch, IFSC, and account holder. The UI explicitly prompts verification (“cross check with passbook/cheque book”), framing this as a high-integrity workflow.

5) “Important Facilities” makes emergency routing actionable

Nearest FRU/Blood Bank/Collection Unit selection is designed as a set-and-sync workflow (including “no nearest selected” states), so the app can support time-sensitive decisions without users hunting through district-wide lists mid-crisis.

RBAC as UX

Permissions show up as “what you can edit” and “what you can only use”

The directory is shared, but not everyone can modify it. The UX makes that visible by scoping entry points and edit actions (especially around committee roles and account details), while keeping calling and viewing consistent.

Boundary 01

Default scope

ASHA starts at village/catchment, ANM at SHC, MO at PHC, but all can drill up/down.

Boundary 02

Governed edits

Add/select flows enforce designation + verification prompts to avoid uncontrolled edits.

Boundary 03

High-integrity data

Account details and “nearest facility” selections behave like official configuration, not casual notes.

Edge cases

Handled the messy stuff on purpose

These are daily realities: missing data, duplicates, partial configuration. Each case is treated as a predictable state so field users don’t get stuck or improvise their own system.

“No Member Added” states

Empty-state handling is explicit across committee and officer lists so users know it’s missing data, not a broken screen.

Constraint addressed: incomplete onboarding and partial governance setup.

Select existing vs add new

Add/select flows support “Already listed” pickers and “Add new” forms to reduce duplicate entries and keep official records stable.

Constraint addressed: duplicates + split ownership of data entry.

“No nearest selected” configuration

Important Facilities clearly shows “No nearest selected” states and routes users into selection flows with save confirmations.

Constraint addressed: incomplete setup and emergency readiness gaps.

Verification prompts for sensitive data

Account and officer/member additions are protected by “check details before saving” confirmations to reduce irreversible mistakes.

Constraint addressed: high-cost errors on official records.

System fit

A shared service across modules: escalation, incentives, and referral readiness

Officer Directory supports the entire platform because it reduces dependency on memory and informal networks. When casework involves follow-ups, referrals, approvals, or payouts, the system needs a stable chain of contact and responsibility.

“My Team” as execution support

Team lists reduce time-to-escalation during exception handling (missed follow-ups, approvals pending, referral coordination).

Important Facilities as operational readiness

FRU and Blood facility configuration supports time-critical decisions and standardizes “nearest” across the system.

Impact

Fewer blind escalations and less dependency on informal networks

  • Hierarchy-first navigation makes escalation predictable across District/Block/PHC/SHC/Village.
  • Role grouping reduces scanning time: users don’t sift through mixed lists.
  • Structured add/select flows reduce duplicates and keep the directory stable over time.
  • Important Facilities configuration makes emergency routing actionable with distance context.

Concrete signals in the UI

The module consistently surfaces counts and constraints (example: 15 district-wide FRUs/Blood Banks/Blood Collection Units, 03 officer group lists, and village-level group counts of 04 per category), showing this is designed as an operational service rather than a static directory page.

What’s next

Working on the full deck: reduce drift and strengthen integrity

Duplicate prevention and merge logic

Add detection for same phone/name across scopes and guide users into “select existing” instead of creating parallel records.

Role-aware “edit history” for governance

Surface who added/updated a member and when, especially for committee and account details, to reduce disputes and support audits.

Make “nearest facility” reusable across modules

Treat nearest FRU/Blood configuration as shared platform settings so referral flows don’t re-ask for the same knowledge repeatedly.

Day-start that grounds work in place, context, and status

A role-aware day-start system that makes punch-in a real operational state, not a polite suggestion. Users select where work is happening (facility, village, outreach, VHND, monitoring, meetings), see execution counters and pending governance work, and get a clean “start visit” path without menu hunting.

Problem

Daily execution breaks when punch-in is optional and context is ambiguous

In multi-program field operations, “logged in” does not mean “ready to work.” Users need a clear daily state (punched-in vs not), a declared work context (facility, village, outreach, VHND, monitoring, meetings), and a surface that shows what matters today. Without that, the UI turns into a maze of menus and retries, and teams fall back to calling supervisors for direction.

What day-start must solve at scale

  • Punch-in as state: prevent “work without context” and make “no visits” explainable.
  • Context selection: capture where work is happening (PHC/SHC/village/outreach/VHND/meetings) for correct reporting and routing.
  • Role-specific surfaces: MO monitoring isn’t ANM VHND, and a pharmacist shouldn’t wade through irrelevant options.
  • Governance visibility: surface verification/approval queues for supervisory roles as first-class work.
  • Offline + sync legibility: make “last synced” and “pending” states visible so users trust the system under weak connectivity.

Users

One day-start, multiple job realities

Day-start supports the same underlying loop (punch-in → work → punch-out) but must reflect role realities. A PHC Medical Officer needs supervisory routes and monitoring contexts. ANM/CHO needs SHC and village contexts. Staff nurses need speed and clarity. Pharmacists and lab teams need the same daily state, but with their own operational surfaces and tools.

Medical Officer (PHC)

Starts at PHC, supervises outreach and VHND, monitors SHCs and villages, and runs meetings. Needs governance tiles (approvals required) and clear “visit-at” contexts to avoid misreporting.

Staff Nurse

Needs a fast punch-in for PHC service delivery and support activities (outreach, VHND). Day-start must keep the next action obvious and not drown the role in supervisory choices.

ANM / CHO

Punch-in spans SHC visits, outreach activity, VHND sessions, and meetings (e.g., VHSNC). Village selection and meeting type must be structured so daily outputs match real work.

Pharmacist / Lab Technician

Same day-start loop, different tools. Needs PHC context plus quick access to role work (patient search, drug stock/inward-outward, lab test execution) without losing the daily state model.

Flow

Punch-in → land → start visit: a predictable loop

The system separates daily state (punch-in / punch-out) from work context (visit-at). Users can confirm where they’re working, see status and counters, and start the next action without navigating across programs.

Routing behavior

  • Role-based options: punch-in choices adapt to posting and role (PHC vs SHC, monitoring vs VHND).
  • Explain empty states: “No visits” becomes a predictable state with a single action: punch-in.
  • Keep the loop visible: punch-in time, working hours, and sync states are accessible without leaving the work surface.

Design decisions

Designed day-start as a state machine, not a home screen

1) Punch-in gate before work

“No visits” is a common reality, not an error. Punch-in establishes daily state so execution and reporting are consistent before users enter tasks.

2) Context cards with plain-language descriptions

Punch-in options are framed as job contexts (PHC care, outreach, VHND, monitoring, meetings) with descriptions, reducing wrong selections and training load.

3) Separate “Visit at” from “Punch-in”

Punch-in sets state. Visit-at sets where work is happening. Keeping them distinct prevents “I punched in so why can’t I start visit?” confusion and preserves reporting integrity.

4) Show execution counters upfront

OPD, vitals, labs, medicines, ABHA, and activity summaries appear as first-glance throughput signals so teams can self-correct without supervisor calls.

5) Surface governance counts for supervisory roles

Verification and approval queues are treated as work, not admin clutter. Putting counts on day-start makes governance visible and routable.

RBAC as UX

Posting, role, and geography shape what “start my day” even means

Day-start is where RBAC becomes tangible. Punch-in options, visit-at choices, and shortcuts are shaped by where a user is posted and what they’re accountable for. Supervisory roles see governance actions and monitoring contexts; frontline roles see fewer, clearer choices.

Boundary 01

Module visibility & shortcuts

Role surfaces show only what the user can act on today, reducing clutter and misclicks.

Boundary 02

Geography + facility mapping

PHC/SHC/village context drives punch-in and visit-at choices so reporting stays correct.

Boundary 03

Governance tiles

Approvals and verifications surface as actionable counts for supervisory roles, not hidden admin pages.

Edge cases

Built for real conditions: no visits, wrong location, pending sync

No Visits is a valid state

Instead of showing an empty dashboard that feels broken, day-start explains why work is blocked and routes to punch-in with one clear action.

Sync pending is made explicit

“Last synced” and “pending” states sit in the utility layer (more actions), reducing duplicate submissions and support calls under weak networks.

Other facility / village selection

Switching between “my facility” and “other” contexts prevents wrong reporting when staff cover multiple SHCs or villages.

Meeting types captured as structured context

Meetings are not one blob. Types like VHSNC, sector, block, and PHC meetings are captured to preserve operational meaning and downstream reporting.

System fit

Day-start connects governance, planning, and registers

Day-start is the bridge between “the system has data” and “a team can execute today.” Work plan and registers provide starting points for field work, while approvals and verifications keep governance visible for supervisory loops.

Work plan and registers

Work plan surfaces “what to do next,” and registers provide direct entry points for day-to-day operations without hunting across program menus.

Approvals required as operational work

Health facility, village/ward, and new user verification counts appear as tiles so supervisory roles can resolve governance before downstream work is impacted.

Impact

A clearer start reduces retries and improves throughput visibility

  • Punch-in makes daily state explicit so “no visits” becomes explainable instead of confusing.
  • Context selection reduces wrong reporting and helps users route into the right work surface faster.
  • Role-aware counters make execution visible without manual follow-ups and constant supervisor coordination.
  • Governance tiles keep approvals and verifications from becoming invisible backlog.

What’s next

What’s next: reduce friction without hiding control

Role-based defaults for punch-in contexts

Keep the full set available, but preselect and prioritize the most common contexts per role and posting to speed up day-start without reducing governance integrity.

Better offline cues and safe recovery paths

Strengthen sync state explanations and add clearer “what happens when offline” messaging so users trust the system under weak connectivity.

Lightweight audit trail for supervisor corrections

When supervisors correct facility/village mappings or approvals, show small “what changed” cues to reduce back-and-forth and speed up resolution.
